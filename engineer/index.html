包管理

pnpm

pnpm相对于传统的包管理工具npm具有更低的内存占用、更快的安装速度和非扁平化的node_modules依赖管理

特性

  • 工作空间（Workspce）

    pnpm 内置了对单一存储库（也称为多包存储库、多项目存储库或单体存储库）的支持。 你可以创建一个工作空间以将多个项目合并到一个仓库中。

    这使得pnpm能够创建monorepo项目

  • 目录（category）
    pnpm 提供了依赖项版本可复用常量定义的协议，可以在package.json中使用定义的常量；能在monorepo项目中统一管理更新依赖

  • 配置项依赖（config dependences） 10.x
    配置项依赖允许在monorepo中分享和中心化管理配置文件、设置和hook

  • 别名
    pnpm支持给依赖配置别名

pnpm 与 npm/yarn 不同点
  • pnpm采用软链接跟硬链接的方式引入依赖
  • pnpm采用非扁平化的node_modules依赖管理
    更多关于pnpm的内容可以看这里

打包工具

该节主要讨论的前端构建/打包工具有: webpack、vite、rollup、esbuild。ps: 后期可能会更新

webpack

webpack 作为前端热门的打包工具，其社区相当完善；接下来主要讨论 webpack 的打包原理以及几个核心的内容: loader、plugin、HMR

HMR (Hot Module Replacement)

HMR 功能会在运用程序运行过程中，可以不用重新加载页面而进行模块的替换、添加或删除。

它的工作原理主要是webpack在webpck-dev-server中创建一个HMR Server与浏览器中的HMR Runtime进行webscoket通讯来实现更新的。那么它的热更新过程如下(如果启用的话):

  • 当启动一个应用程序的时候，webpack会生成一个manifest(JSON)文件，该文件包含了complication hash 和所有module的列表；同时webpack会将HMR Runtime注入到生成的build.js文件中去
  • 当你更改文件的时候，会被webpack检测到；complier会构建应用程序同时生成新的manifest文件并且与老的文件进行对比，这一过程会生成hot update(updated chunks，包含在manifest文件中)
  • 这个hot update会被发送到HMR Server去，然后由HMR Server将更新发送到HMR Runtime
  • HMR Runtime不会去打包这些hot update而是使用具体的loader去控制这些更新(比如 CSS 变化会调用 css-loader 或者 style-loader) HMR 被定义的方式
  • 首先要在webpack配置中添加hot: true
  • 然后通过使用module.hotAPI 来使用 HMR 请求

esbuild

esbuild是使用Go语言去编写的，相对于webpack、rollup等构建打包工具有着更快的打包速度；这里主要讲述esbuild中常使用的API、友好的文件类型和plugins功能

特性

esbuild通过API进行构建，主要有三个build、transfer和context常用API；

  • build能够支持源码转换打包的功能，主要使用在构建打包的时候；
  • transform主要支持源码的转换成相应的字符串，主要用于将esbuild作为中间层给其他打包工具源码转换功能，比如转换TypeScript
  • context能够提供相同的构建配置进行多次构建，常用于本地服务器/文件更新等场景

esbuild还天然支持treeshake和minifiy；并且对TypeScript和JSX友好，能够支持TypeScript和JSX转换JavaScript

APIs

较常用的esbuild api有build和transform这两个;esbuild apis可以在CLI/JS/GO三种环境中运行，下面将主要介绍CLI和JS两种环境中的使用

build

buildapi主要使用在根据一个或者多个入口文件将文件打包成一个或者多个bundle的情况，同时esbuild会将这些bundle写入到文件系统中，并且该api支持多种配置；使用方式如下：

  • CLI
  • js build api的高级用法涉及设置长期使用的构建内容；该内容在js和go中是明确的对象，在CLI中是隐式的；在给定的内容中，会共享一样的构建配置；一系列的构建会逐步的完成

有三种不同的增量构建api

  • watch mode: 会告诉esbuild监听文件的变化去自动重新构建你的文件

    • CLI
    • js

  • server mode: esbuild会开启一个本地开发服务器，当有请求过来的时候会开启一个新的构建以确保你的app总是最新的

    • CLI
    • js

  • Rebuild mode: 让你手动调用构建

polyfill

polyfill是为了让旧浏览器或者环境兼容支持新特性/标准特性；而Babel作为广泛使用的一个端到端的javascript编译器，能够很好的作为polyfill的工具

Babel

Babel是一个被广泛使用的javascript编译器，是一组用来进行多种形式的静态分析的模块

静态分析是在没有执行代码的情况下代码分析的一个过程（在运行中分析代码的代码分析叫做动态分析）。静态分析的目的有很多，可以用于linting、编译、代码高亮、代码转换、优化、最小化等

Babel作为一个端到端的javascript编译器，通常被称作转换器；这就说明可以通过Babel将代码进行转换后输出新的代码。Babel的内容

Modules

在很多的语言中都有着模块的概念；并且开发者们一直尝试使用不同的模块系统来提供更有组织、更具协作性的工作环境。那么什么是模块呢？模块就是有独立的作用域，并且能够通过对外暴露变量和属性来提供代码复用。在javascript中可以理解为一个文件就是一个模块。目前在javascript中比较热门的模块化规范/模块系统有，CommonJS、AMD、UMD、ESM和CMD。下面将讨论和对比一下这些模块化规范

CommonJS（CJS）

CommonJS是被Node.js选用和使用在服务端的模块化规范；它允许通过require和module.exports来导入和导出模块。这能让模块同步的加载并且提供了简单的语法

CommonJS采用同步加载的方式去引用模块是因为在服务端中读取本地磁盘的速度会比通过网络获取的速度要快；同步加载方案在服务端更适用，所以很多的用于服务端的框架和库更喜欢使用CommonJS

CommonJS的语法

在CommonJS中其实除了使用module.exports导出以外，还可以使用exports（不推荐）来导出，那么两个有什么区别呢

  • module.exports
    在CommonJS中每一个模块都定义在自己的作用域里面，只有显示的暴露模块中的方法或者变量才能被外部模块使用；而module.exports就是允许向外暴露方法的一个特殊对象
  • exports
    exports其实就是module.exports的一个引用对象；所以只能通过.的语法来添加属性暴露模块的内容了；而module.exports可以通过赋值新对象来暴露更多的方法和变量
CommoJS运用于服务端的优势
  • CommonJS被Node.js采用，并且在Node.js项目中被广泛应用
  • CommonJS为服务端应用提供了简单和同步的模块加载
  • CommonJS为Node.js生态系统提供了良好的集成和兼容性

AMD（Asynchronous Module Definition）

AMD模块化规范是为了客户端设计的；它能够支持异步的方式加载模块，这样提升了应用在浏览器中的运行的性能。AMD能够通过require和define的方式来导入和导出模块

require.js作为AMD常用的库

AMD的语法

通过define()函数来定义，并且需要使用到其他的依赖时，可以将依赖配置在[]中作为define()的第一个参数

AMD作为客户端的规范的优势
  • AMD为客户端提供了异步加载模块的能力
  • AMD为大型应用提供了性能和可用性的优势
  • AMD为依赖管理和模块定义提供了可延展的结构

与AMD规范类似的还有CMD规范(以sea.js为主要代表)，它们的不同点在于AMD 推崇依赖前置、提前执行；CMD 推崇依赖后置、延迟执行

ESM（ECMAScript Module）

ESM被认为是javascript的官方模块化规范，并且它在现代浏览器中原声支持；ESM可以通过import和export关键字来实现导入和导出。这样为静态分析和tree shaking提供了更多优化的机会

ESM也提供了模块的异步加载的支持

ESM的语法

ESM模块不是对象，import命令会被javascript引擎静态解析，在编译时就引进了模块代码，而不是在代码运行时加载；正因为这样，使得静态分析变得可能

ESM与CommonJS的差异
1、ESM模块输出的是值的引用，CommonJS模块输出的是值的拷贝
  • CommonJS输出的是值的引用，即一旦输出值后模块内部的变化是不会影响外部引用的
  • ESM的运行机制与CommonJS的运行机制是不一样的。ESM对脚本静态分析的时候，解析到模块脚本中的import，就会生成一个只读引用；等脚本运行时在跟进引用去已加载的模块中取值，这跟符号链接很像。因此ESM模块是动态引用，并不会缓存值，模块里面的变量绑定其所在的模块。
2、CommonJS模块是运行时加载，ESM模块是编译时输出
  • CommonJS模块就是对象，在输出时先加载模块，并且生成一个对象，然后运行时从这个对象中获取方法，这种就是运行时加载
  • ESM模块不是对象，而是通过export显示指定输出的代码，然后import时采用静态的形式使用；这个过程都是在编译阶段静态分析的时候完成的，这就是编译时加载
选用ESM的优势
  • ESM是现代javascript的官方规范，并且是前瞻性的选择
  • ESM由浏览器原生支持，不用额外安装库
  • ESM能够支持静态检查和tree shaking这样的代码优化
  • ESM通过支持模块异步加载提供了更好的性能

UMD（Universal Module Definition）

UMD提供了支持CommonJS和ESM的通用结构；这样能够运行模块运行在不同的环境中。UMD能够适用于运行在浏览器和服务端的应用程序

UMD的语法
选用UMD的优势
  • UMD能够支持CommonJS和ESM，提供了通用结构
  • UMD提供了能够工作在不同平台的能力
  • UMD具有吸引广泛用户群的优势

总结

  • CommonJS/ESM/ADM/UMD是模块化规范，对应的实现分别是Node.js/JavaScript/require.js/通用结构
  • CommonJS是针对于服务端的规范；ESM、AMD、CMD是针对于客户端的规范；UMD是适用于服务端和客户端的通用解决方案
  • AMD与CMD的区别：
    • AMD是预加载，即在js资源下载完成后同步去执行模块；CMD是懒加载，即在js资源下载完成后不去执行，当执行到对应的代码再去加载模块。
  • CommonJS于ESM的区别：
    • CommonJS模块输出的是值的拷贝；ESM模块输出的是值的引用

设计模式

发布-订阅模式

发布-订阅模式就是通过中间者管理发布者和订阅者之间的关系，而这个中间者通常通过管理一些事件来进行消息的订阅和发布

观察者模式

观察者模式是观察者与被观察者之间直接进行通信，只需要在被观察者内部维护一系列的观察者就行了
